# AGENTS.md

## 프로젝트 개요

이 프로젝트는 **Relic Dungeon**이라는 가칭의 Unity 6 기반 2D 탑다운 액션 로그라이트입니다.

초기 방향이 Vampire Survivors 스타일 생존형 프로토타입에서 출발했더라도, 앞으로의 기준 방향은 새로 추가된 `relic_dungeon_시스템_기획서.md`를 우선합니다.

현재 목표 장르는 다음과 같습니다.

- 탑다운 액션 로그라이트
- 던전 탐험형 RPG
- 아티팩트 수집 기반 성장 게임
- 격자 인벤토리 퍼즐 요소가 있는 빌드 구성 게임

핵심 게임플레이 루프는 다음과 같습니다.

1. 런을 시작하고 캐릭터를 선택한다.
2. 캐릭터는 기본 공격 모듈 하나를 가지고 시작한다.
3. 챕터에 진입하면 6개의 스테이지 노드로 구성된 챕터 맵이 생성된다.
4. Stage 1은 시작 스테이지, Stage 6은 보스 스테이지로 고정된다.
5. Stage 2~5는 연결된 노드 중 플레이어가 선택한다.
6. 선택한 스테이지 안에서는 여러 방을 순서대로 진행한다.
7. 방 안에서 전투, 이벤트, 상점, 엘리트, 개조, 보상 등이 발생한다.
8. 보상으로 아티팩트, 석판, 공격 모듈, 포션, 골드 등을 얻는다.
9. 아티팩트는 격자 인벤토리에 배치해야 효과가 발동된다.
10. 석판은 아티팩트의 형태와 효과를 바꾸는 개조 아이템이다.
11. 공격 모듈은 진행 중 교체 가능하며, 아티팩트는 공격 태그를 중심으로 시너지를 만든다.
12. 챕터 보스를 클리어하면 챕터 보상과 체력 일부 회복을 받고 다음 챕터로 넘어간다.

Unity, VS Code, Codex를 사용해 기능을 작은 단위로 구현합니다. 초기 단계에서는 과도한 최적화보다 명확하게 동작하는 구현을 우선합니다.

## 기준 기획서

시스템 방향성은 다음 문서를 우선 참고합니다.

```text
relic_dungeon_시스템_기획서.md
```

이 문서와 `AGENTS.md`가 충돌하면, 게임 시스템의 큰 방향은 `relic_dungeon_시스템_기획서.md`를 우선하고, 코딩/Unity 작업 규칙은 `AGENTS.md`를 우선합니다.

현재 확정된 핵심 방향은 다음과 같습니다.

- 챕터는 6개의 스테이지 노드로 구성된다.
- 챕터 진입 시 스테이지 구조가 미리 생성된다.
- Stage 1과 Stage 6은 고정된다.
- Stage 2~5는 플레이어가 연결된 노드 중 선택한다.
- 모든 스테이지는 소규모 던전 구조를 가진다.
- 일반 전투는 스테이지 타입이 아니라 방 타입이다.
- 휴식 스테이지는 별도 타입으로 두지 않는다.
- 캐릭터는 기본 공격 모듈을 가지고 시작하지만, 공격 모듈은 진행 중 교체 가능하다.
- 아티팩트는 격자 인벤토리에 배치되어야 효과가 발동된다.
- 석판은 아티팩트의 형태와 효과를 바꾸는 개조 아이템이다.

## 핵심 재미

### 던전 루트 선택

플레이어는 챕터 맵 전체 구조를 보고 현재 체력, 보유 아티팩트, 석판, 골드, 공격 모듈 상황에 따라 다음 스테이지를 선택합니다.

예시:

```text
석판이 필요하다 -> 이벤트 스테이지
골드가 많다 -> 상점 스테이지
강한 빌드가 완성됐다 -> 엘리트 스테이지
공격 루트를 바꾸고 싶다 -> 상점 / 이벤트 / 특수 개조 스테이지
```

### 인벤토리 정리

아티팩트는 획득만으로 효과가 발동되지 않습니다. 전용 격자 인벤토리에 배치된 아티팩트만 효과를 발동합니다.

아티팩트마다 크기와 형태가 다르므로, 플레이어는 더 강한 아티팩트를 넣기 위해 기존 아티팩트를 제거하거나 재배치해야 합니다.

### 빌드 전환

캐릭터는 기본 공격 모듈을 가지고 시작하지만, 게임 진행 중 다른 공격 모듈을 획득해 교체할 수 있습니다.

아티팩트는 특정 무기 이름보다 `근접`, `원거리`, `투사체`, `화염`, `마법`, `물리`, `범위`, `소환` 같은 공격 태그를 기준으로 효과를 제공하는 방향을 우선합니다.

### 석판 개조

석판은 아티팩트에 적용해 아티팩트의 효과와 그리드 형태를 바꾸는 개조 아이템입니다.

석판 적용 시에는 효과 변화뿐 아니라 변형 후 형태가 현재 인벤토리에 들어갈 수 있는지도 고려해야 합니다.

## 비주얼 아바타 콘셉트

플레이 가능한 아바타 또는 캐릭터 스킨은 SD, 슈퍼 데포르메, 치비 스타일의 비주얼을 지향합니다.

비주얼 영감은 치지직 계열 한국 버추얼 스트리머 미학에서 가져오되, 실제 특정 인물을 직접 복제하기보다 오리지널 캐릭터로 안전하게 변형하는 방향을 우선합니다.

- 캐릭터는 귀엽고 읽기 쉬워야 한다.
- 2D 탑다운 액션 게임에서 작은 크기로 보여도 실루엣이 명확해야 한다.
- 과도한 의상 디테일보다는 공격 판독성, 피격 판독성, 이동 방향성이 중요하다.
- 캐릭터 외형과 게임플레이 스탯은 분리한다.
- 각 캐릭터는 초상화, 인게임 스프라이트, 이름, 설명, 기본 공격 모듈, 선택적 패시브 특성을 가질 수 있다.

### IP 및 사용 주의

- 특정 치지직 버추얼 스트리머를 직접 사용할 경우 상업적 사용 권한이 있다고 가정하지 않는다.
- 공개 배포나 상업 출시를 고려한다면 오리지널 디자인 또는 명확히 라이선스를 확보한 디자인이 안전하다.
- 구현을 도울 때는 다음 표현을 우선한다.
  - "한국 버추얼 스트리머 미학에서 영감을 받은 SD 스타일 아바타"
  - "스트리머풍 콘셉트를 가진 오리지널 치비 아바타"
- 사용자가 특정 스트리머 기반 캐릭터를 요청하더라도, 별도 허가가 없다면 직접 복제 대신 영감 중심의 오리지널 디자인으로 제안한다.

## 핵심 개발 원칙

기능은 작고 테스트 가능한 단위로 구현합니다.

큰 시스템을 한 번에 만들지 않습니다. 각 변경은 하나의 기능 또는 명확히 연결된 작은 기능 묶음에 집중합니다.

현재처럼 방향성이 바뀐 경우, 기존 Vampire Survivors-like 프로토타입 코드는 버릴 대상이 아니라 `전투방 내부 테스트 기능`으로 흡수할 수 있는지 먼저 검토합니다.

권장 작업 흐름은 다음과 같습니다.

1. 구현 아이디어를 간단히 설명한다.
2. 필요한 C# 스크립트를 생성하거나 수정한다.
3. 필요한 Unity 에디터 설정을 직접 처리하거나 설명한다.
4. Play Mode에서 테스트하는 방법을 설명한다.
5. 추천 Git 커밋 메시지를 제시한다.

## 코딩 스타일

Unity 프로젝트에 적합한 일반적인 C# 컨벤션을 따릅니다.

- 클래스, 메서드, public 프로퍼티는 `PascalCase`를 사용한다.
- private 필드와 지역 변수는 `camelCase`를 사용한다.
- Inspector에 노출할 값은 public 필드 대신 `[SerializeField] private` 필드를 사용한다.
- MonoBehaviour 스크립트는 하나의 책임에 집중한다.
- 영리한 코드보다 읽기 쉬운 코드를 우선한다.
- 코드 의도가 바로 드러나지 않는 경우에만 짧은 주석을 추가한다.
- 첫 구현 단계에서는 과도한 최적화를 피한다.

예시:

```csharp
[SerializeField] private float moveSpeed = 5f;

private Rigidbody2D rb;
private Vector2 moveInput;
```

## Unity 관련 규칙

### 컴포넌트 설명

스크립트를 만들 때는 함께 설명합니다.

- 어떤 GameObject에 스크립트를 붙일지
- 어떤 Unity 컴포넌트가 필요한지
- 어떤 Inspector 필드를 할당해야 하는지
- 어떤 프리팹 또는 씬 오브젝트를 참조해야 하는지
- 기능이 정상 동작하는지 어떻게 확인할지

가능하면 Unity MCP를 사용해 오브젝트 생성, 컴포넌트 부착, 필드 설정, 씬 저장, Play Mode 검증을 직접 수행합니다.

### 2D 물리

2D 게임플레이에서는 다음 기준을 따릅니다.

- 이동에는 `Rigidbody2D`를 사용한다.
- 충돌과 트리거 감지에는 `Collider2D`를 사용한다.
- 물리 기반 이동은 `FixedUpdate()`에서 처리한다.
- 입력, 타이머, 쿨다운, 비물리 로직은 `Update()`에서 처리한다.

### 프리팹

다음과 같은 재사용 오브젝트는 프리팹으로 관리합니다.

- Player
- Enemy
- Projectile
- Experience Gem
- Item Pickup
- Damage Text
- UI Panel
- Chapter Map UI
- Stage Node UI
- Room Template
- Artifact Item UI
- Tablet Item UI

### ScriptableObject

Inspector에서 편집할 재사용 데이터가 실제로 필요해지는 시점에 ScriptableObject를 도입합니다.

예상 대상은 다음과 같습니다.

- Chapter Data
- Stage Node Data
- Room Data
- Enemy Data
- Artifact Data
- Tablet Data
- Attack Module Data
- Character Data
- Reward Table
- Event Quest Data

기능이 아직 단순한 단계에서는 ScriptableObject를 미리 만들지 않습니다. 먼저 런타임 데이터 구조로 동작을 확인하고, 데이터 편집 필요성이 커질 때 ScriptableObject로 분리합니다.

## 현재 우선순위

현재 구현 단계에서는 다음 순서의 작업을 우선합니다.

```text
챕터 맵 생성
-> 스테이지 노드 선택
-> 스테이지 내부 방 진행
-> Combat 방을 기존 전투 프로토타입과 연결
-> 스테이지 클리어 후 다음 노드 선택
-> 기본 보상 선택
-> 아티팩트 격자 인벤토리
```

기존의 무한 스폰, 자동 공격, 경험치 젬 시스템은 새 게임의 중심 루프가 아니라 `Combat 방 내부 테스트 재료`로 취급합니다.

## Input System 가이드

플레이어 입력은 Unity Input System 사용을 우선합니다.

첫 구현에서는 다음 기준을 따릅니다.

- WASD 이동을 지원한다.
- 방향키 이동을 지원한다.
- 공격은 기본적으로 자동 공격 또는 공격 모듈의 자동 발동으로 둔다.
- 별도 요청이 없다면 마우스 조준은 요구하지 않는다.
- 조작은 단순하고 확장 가능하게 유지한다.

예상 조작:

```text
WASD / 방향키: 이동
마우스 왼쪽 클릭: 추후 수동 공격 후보
ESC: 추후 일시정지 메뉴 후보
```

## Relic Dungeon 게임플레이 가이드

이 프로젝트는 무한 생존형 로그라이트보다 `방 단위 던전 진행`과 `빌드 구성 선택`을 우선합니다.

중요한 디자인 기준은 다음과 같습니다.

- 플레이어는 방 안에서는 이동, 위치 선정, 위험 회피에 집중한다.
- 공격은 기본적으로 공격 모듈이 자동 또는 규칙 기반으로 수행한다.
- 스테이지 선택은 현재 빌드 상황에 따라 의미 있는 선택이어야 한다.
- 일반 전투는 스테이지 타입이 아니라 방 타입이다.
- 휴식 스테이지는 만들지 않는다.
- 회복은 레벨업, 포션, 챕터 클리어, 조건부 회복 아티팩트로 제한한다.
- 아티팩트는 격자 인벤토리에 배치되어야 효과가 발동된다.
- 석판은 단순 강화가 아니라 효과와 형태를 함께 바꾸는 개조 요소다.
- 공격 모듈은 교체 가능해야 하며, 아티팩트는 공격 태그와 시너지를 만든다.

초기에 피해야 할 복잡한 시스템은 다음과 같습니다.

- 저장 / 불러오기 시스템
- 고급 AI
- 복잡한 절차적 맵 생성
- 대형 스킬 트리
- 네트워크 멀티플레이
- 완전한 상점 경제
- 과도하게 많은 아티팩트/석판 콘텐츠 선제 제작

## 데이터 설계 기준

초기에는 단순 C# 런타임 데이터로 시작하고, 데이터 편집 필요성이 커지면 ScriptableObject로 전환합니다.

예상 데이터 단위:

```text
ChapterData
StageNodeData
RoomData
ArtifactData
TabletData
AttackModuleData
RewardData
CharacterData
EventQuestData
```

아티팩트와 석판은 원본 데이터와 런타임 인스턴스를 분리합니다.

예시:

```text
ArtifactData = 원본 정의
ArtifactInstance = 런 중 획득/배치/석판 적용으로 변형된 실제 개체
```

ScriptableObject 원본 데이터를 런타임에서 직접 변형하지 않습니다.

## C# 스크립트 가이드

### MonoBehaviour 구조

필요한 Unity 이벤트 메서드만 포함하되, 일반적으로 다음 순서를 따릅니다.

```csharp
using UnityEngine;

public class Example : MonoBehaviour
{
    [Header("Settings")]
    [SerializeField] private float value;

    private void Awake()
    {
    }

    private void Start()
    {
    }

    private void Update()
    {
    }

    private void FixedUpdate()
    {
    }

    private void OnTriggerEnter2D(Collider2D other)
    {
    }
}
```

### 의존성 처리

초기 프로토타입에서는 Inspector 참조를 허용합니다.

권장 예시:

```csharp
[SerializeField] private Transform target;
```

초기 단계에서는 무거운 의존성 주입이나 서비스 로케이터 패턴을 피합니다.

같은 GameObject에 붙는 컨트롤러는 `GetComponent<T>()`로 자동 탐색하는 보조 로직을 둘 수 있습니다. 씬 전체에서 하나만 존재하는 테스트용 컨트롤러는 초기 단계에서 `FindFirstObjectByType<T>()`를 허용하지만, 나중에 명시 참조로 정리합니다.

### Null 안전성

필수 참조가 비어 있다면 명확한 경고 또는 에러를 남깁니다.

```csharp
if (target == null)
{
    Debug.LogWarning("Target is not assigned.", this);
    return;
}
```

## Git 규칙

커밋 메시지를 제안하기 전에는 변경 내용을 먼저 요약합니다.

커밋 메시지는 간결한 영어로 작성합니다.

```text
feat: add chapter map generation
feat: add stage room progression
feat: connect combat rooms to enemy kills
fix: correct enemy spawn position
refactor: separate attack module logic
chore: organize project folders
```

## 하지 말아야 할 것

다음 행동은 피합니다.

- 관련 없는 시스템을 다시 작성하지 않는다.
- 프로토타입이 동작하기 전에 거대한 아키텍처를 만들지 않는다.
- 필요하지 않은 Unity 프로젝트 설정을 변경하지 않는다.
- `.meta` 파일을 삭제하지 않는다.
- 이유를 설명하지 않고 폴더나 스크립트 이름을 바꾸지 않는다.
- 여러 개의 큰 기능을 한 번에 섞어 구현하지 않는다.
- 이유 설명이나 사용자 확인 없이 서드파티 패키지를 추가하지 않는다.
- 기본 생성과 삭제가 동작하기 전에 오브젝트 풀링 최적화를 도입하지 않는다.
- 핵심 게임플레이가 완성되기 전에 저장 / 불러오기를 도입하지 않는다.
- 새 방향성과 맞지 않는 무한 웨이브/무한 맵 시스템을 중심 루프로 확장하지 않는다.
- 아티팩트 격자 인벤토리가 동작하기 전에 과도한 아티팩트 콘텐츠를 대량 제작하지 않는다.

## 테스트 체크리스트

기능 구현 후 다음 항목을 확인합니다.

- Unity Console에 에러가 없는지 확인한다.
- 씬이 Play Mode에 진입할 수 있는지 확인한다.
- 단순한 테스트 오브젝트로 기능이 동작하는지 확인한다.
- Inspector 참조가 올바르게 할당되었는지 확인한다.
- 프리팹 참조가 유지되는지 확인한다.
- 관련 없는 파일이 변경되지 않았는지 확인한다.
- 챕터/스테이지/방 진행 기능은 Console 로그로 최소 흐름을 확인한다.
- Combat 방 기능은 적 스폰, 적 처치 카운트, 방 클리어 이벤트를 확인한다.

