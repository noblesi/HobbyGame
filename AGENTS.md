# AGENTS.md

## 프로젝트 개요

이 프로젝트는 Vampire Survivors 스타일의 플레이 감각을 참고한 Unity 2D 탑다운 로그라이트 액션 게임입니다.

Unity, VS Code, Codex를 사용해 기능을 작은 단위로 하나씩 구현합니다. 초기 단계에서는 과도한 최적화보다 명확하게 동작하는 구현을 우선합니다.

주요 게임플레이 루프는 다음과 같습니다.

1. 플레이어가 2D 맵을 이동한다.
2. 적이 계속 생성된다.
3. 적이 플레이어를 추적한다.
4. 플레이어는 자동으로 공격한다.
5. 처치된 적은 경험치 또는 아이템을 떨어뜨린다.
6. 플레이어는 레벨업하고 업그레이드를 선택한다.
7. 시간이 지날수록 웨이브 난도가 상승한다.
8. 플레이어가 사망하거나 목표를 클리어하면 런이 종료된다.

## 비주얼 아바타 콘셉트

플레이 가능한 아바타 또는 캐릭터 스킨은 SD, 슈퍼 데포르메, 치비 스타일의 비주얼을 지향합니다.

비주얼 영감은 치지직 계열 한국 버추얼 스트리머 미학에서 가져오되, 실제 특정 인물을 직접 복제하기보다 오리지널 캐릭터로 안전하게 변형하는 방향을 우선합니다.

- 캐릭터는 귀엽고 읽기 쉬워야 한다.
- 2D 탑다운 액션 게임에서 작은 크기로 보여도 실루엣이 명확해야 한다.
- 과도한 의상 디테일보다는 공격 판독성, 피격 판독성, 이동 방향성이 중요하다.

### 디자인 방향

캐릭터 비주얼이나 관련 시스템을 제안할 때는 다음 기준을 따릅니다.

- SD / 치비 체형을 우선한다.
- 머리 대 몸 비율을 크게 잡아 작은 인게임 크기에서도 잘 보이게 한다.
- 색상, 헤어스타일, 액세서리, 표정과 분위기로 아바타의 개성을 살린다.
- 작은 화면에서도 읽히도록 디테일을 단순화한다.
- idle, move, hit, attack, death 애니메이션을 지원할 수 있는 스프라이트 구조를 고려한다.

### IP 및 사용 주의

- 특정 치지직 버추얼 스트리머를 직접 사용할 경우 상업적 사용 권한이 있다고 가정하지 않는다.
- 공개 배포나 상업 출시를 고려한다면 오리지널 디자인 또는 명확히 라이선스를 확보한 디자인이 안전하다.
- 구현을 도울 때는 다음 표현을 우선한다.
  - "한국 버추얼 스트리머 미학에서 영감을 받은 SD 스타일 아바타"
  - "스트리머풍 콘셉트를 가진 오리지널 치비 아바타"
- 사용자가 특정 스트리머 기반 캐릭터를 요청하더라도, 별도 허가가 없다면 직접 복제 대신 영감 중심의 오리지널 디자인으로 제안한다.

### 구현 방향

아바타 관련 시스템은 다음 순서로 접근합니다.

1. 먼저 플레이스홀더 스프라이트로 게임플레이를 구현한다.
2. 이후 스킨을 교체할 수 있도록 캐릭터 비주얼 구조를 분리한다.
3. 게임플레이 스탯과 캐릭터 외형 데이터를 분리한다.
4. 각 아바타는 다음 데이터를 가질 수 있게 한다.
   - 초상화
   - 인게임 스프라이트
   - 이름
   - 설명
   - 선택적 패시브 특성

향후 권장 폴더 구조는 다음과 같습니다.

```text
Assets
└─ _Project
   ├─ Art
   │  └─ Characters
   │     ├─ Avatar_01
   │     ├─ Avatar_02
   │     └─ Common
   ├─ Prefabs
   │  └─ Characters
   └─ ScriptableObjects
      └─ Characters
```

## 코딩 스타일

Unity 프로젝트에 적합한 일반적인 C# 컨벤션을 따릅니다.

### 기본 규칙

- 클래스, 메서드, public 프로퍼티는 `PascalCase`를 사용한다.
- private 필드와 지역 변수는 `camelCase`를 사용한다.
- Inspector에 노출할 값은 public 필드 대신 `[SerializeField] private` 필드를 사용한다.
- MonoBehaviour 스크립트는 하나의 책임에 집중한다.
- 영리한 코드보다 읽기 쉬운 코드를 우선한다.
- 코드 의도가 바로 드러나지 않는 경우에만 짧은 주석을 추가한다.
- 첫 구현 단계에서는 과도한 최적화를 피한다.

예시:

```csharp
[SerializeField] private float moveSpeed = 5f;

private Rigidbody2D rb;
private Vector2 moveInput;
```

## Unity 관련 규칙

### 컴포넌트 설명

스크립트를 만들 때는 함께 설명합니다.

- 어떤 GameObject에 스크립트를 붙일지
- 어떤 Unity 컴포넌트가 필요한지
- 어떤 Inspector 필드를 할당해야 하는지
- 어떤 프리팹 또는 씬 오브젝트를 참조해야 하는지
- 기능이 정상 동작하는지 어떻게 확인할지

### 2D 물리

2D 게임플레이에서는 다음 기준을 따릅니다.

- 이동에는 `Rigidbody2D`를 사용한다.
- 충돌과 트리거 감지에는 `Collider2D`를 사용한다.
- 물리 기반 이동은 `FixedUpdate()`에서 처리한다.
- 입력, 타이머, 쿨다운, 비물리 로직은 `Update()`에서 처리한다.

### 프리팹

다음과 같은 재사용 오브젝트는 프리팹으로 관리합니다.

- Player
- Enemy
- Projectile
- Experience Gem
- Item Pickup
- Damage Text
- UI Panel

### ScriptableObject

Inspector에서 편집할 재사용 데이터가 실제로 필요해지는 시점에 ScriptableObject를 도입합니다.

예상 대상은 다음과 같습니다.

- Weapon Data
- Enemy Data
- Upgrade Data
- Artifact Data
- Character Stats

기능이 아직 단순한 단계에서는 ScriptableObject를 미리 만들지 않습니다.

## Input System 가이드

플레이어 입력은 Unity Input System 사용을 우선합니다.

첫 구현에서는 다음 기준을 따릅니다.

- WASD 이동을 지원한다.
- 방향키 이동을 지원한다.
- 공격은 자동 공격을 기본으로 둔다.
- 별도 요청이 없다면 마우스 조준은 요구하지 않는다.
- 조작은 단순하고 확장 가능하게 유지한다.

예상 조작:

```text
WASD / 방향키: 이동
마우스 왼쪽 클릭: 추후 수동 공격 후보
ESC: 추후 일시정지 메뉴 후보
```

## Vampire Survivors 스타일 게임플레이 가이드

이 프로젝트는 생존형 로그라이트의 감각을 목표로 합니다.

중요한 디자인 기준은 다음과 같습니다.

- 플레이어는 주로 이동과 위치 선정에 집중한다.
- 공격은 보통 자동으로 이루어진다.
- 업그레이드는 체감 가능한 변화를 만들어야 한다.
- 시간이 지날수록 적 압박이 증가해야 한다.
- 초기 시스템은 단순하지만 확장 가능해야 한다.

초기에 피해야 할 복잡한 시스템은 다음과 같습니다.

- 전체 인벤토리
- 저장 / 불러오기 시스템
- 고급 AI
- 복잡한 절차적 맵 생성
- 대형 스킬 트리
- 네트워크 멀티플레이

## C# 스크립트 가이드

### MonoBehaviour 구조

필요한 Unity 이벤트 메서드만 포함하되, 일반적으로 다음 순서를 따릅니다.

```csharp
using UnityEngine;

public class Example : MonoBehaviour
{
    [Header("Settings")]
    [SerializeField] private float value;

    private void Awake()
    {
    }

    private void Start()
    {
    }

    private void Update()
    {
    }

    private void FixedUpdate()
    {
    }

    private void OnTriggerEnter2D(Collider2D other)
    {
    }
}
```

### 의존성 처리

초기 프로토타입에서는 Inspector 참조를 허용합니다.

권장 예시:

```csharp
[SerializeField] private Transform target;
```

초기 단계에서는 무거운 의존성 주입이나 서비스 로케이터 패턴을 피합니다.

### Null 안전성

필수 참조가 비어 있다면 명확한 경고 또는 에러를 남깁니다.

```csharp
if (target == null)
{
    Debug.LogWarning("Target is not assigned.", this);
    return;
}
```

## Git 규칙

커밋 메시지를 제안하기 전에는 변경 내용을 먼저 요약합니다.

커밋 메시지는 간결한 영어로 작성합니다.

```text
feat: add player movement
fix: correct enemy spawn position
refactor: separate weapon cooldown logic
chore: organize project folders
```

## 하지 말아야 할 것

다음 행동은 피합니다.

- 관련 없는 시스템을 다시 작성하지 않는다.
- 프로토타입이 동작하기 전에 거대한 아키텍처를 만들지 않는다.
- 필요하지 않은 Unity 프로젝트 설정을 변경하지 않는다.
- `.meta` 파일을 삭제하지 않는다.
- 이유를 설명하지 않고 폴더나 스크립트 이름을 바꾸지 않는다.
- 여러 개의 큰 기능을 한 번에 섞어 구현하지 않는다.
- 이유 설명이나 사용자 확인 없이 서드파티 패키지를 추가하지 않는다.
- 기본 생성과 삭제가 동작하기 전에 오브젝트 풀링 최적화를 도입하지 않는다.
- 핵심 게임플레이가 완성되기 전에 저장 / 불러오기를 도입하지 않는다.
